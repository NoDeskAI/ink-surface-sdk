# InkLoop 前端标注链路技术文档

> 代码基线：`main`，2026-06-22。  
> 阅读目标：让第一次接触这个项目的人，能从“怎么跑”理解到“每一笔标注怎样变成 AI 旁注”。

## 1. 先把项目跑起来

### 1.1 环境

这是一个 Vite + TypeScript 单页应用。入口文件是 [index.html](../index.html)，业务入口是 [src/main.ts](../src/main.ts)。

当前机器已验证：

```bash
node -v   # v24.14.0
npm -v    # 11.9.0
```

### 1.2 安装和启动

```bash
npm install
cp .env.example .env
npm run dev -- --host 127.0.0.1
```

打开：

```text
http://127.0.0.1:8765/
```

端口固定在 [vite.config.ts](../vite.config.ts)：

```ts
server: { port: 8765, strictPort: true }
```

如果 8765 被占用，Vite 会直接失败，不会自动换端口。

### 1.3 没有 AI key 时能不能用

能打开页面，也能导入 PDF、渲染、画笔迹、测试部分本地几何逻辑。

但这些功能需要 `.env` 里的 `LLM_GATEWAY_KEY`：

- 手写识别：`/api/interpret`
- 局部 OCR / 图片转写：`/api/ocr-vlm`
- 图像解读：`/api/explain-image`
- AI 旁注和答问：`/api/chat`
- AI 重排：`/api/reflow-ai-stream`、`/api/reflow-ai`

没有 key 时，相关请求会抛 `LLM_GATEWAY_KEY 未配置`，主流程会尽量降级，不让页面崩掉。

### 1.4 第一次怎么试

1. 进入 `http://127.0.0.1:8765/`。
2. 点击「导入 PDF」，选择一个有文本层的数字版 PDF。
3. 用钢笔工具圈一个词，或划一行。
4. 默认长停顿综合阈值是 90 秒。为了测试，可在左侧「设置」页把它调到 10 秒。
5. 在页边写一个问题，系统会尝试识别手写并判断是否要立刻回答。
6. 点击顶部「重排」，切到重排阅读视图，再在重排文本上圈画。

左侧导航壳由 [src/dev/console.ts](../src/dev/console.ts) 创建，按 `m` 可折叠或展开。

## 2. 一句话理解这个项目

用户不是在“问一个聊天机器人”，而是在“用笔在原文上表达注意力”。

所以系统要做三件事：

1. 采集：把用户在页面上画的每一笔完整记录下来。
2. 取证：判断这笔画在哪里、标到了哪些原文对象、是否包含手写或图片。
3. 回屏：把 AI 回答贴回原文现场，而不是只显示在聊天框里。

最核心的数据流是：

```text
PointerEvent
→ Stroke
→ AnnotationEvent
→ Mark
→ HMP
→ Session
→ MarkGraph
→ InferenceView
→ /api/chat
→ ScreenOverlay
→ whisper / reader note
→ IndexedDB
```

如果只记住一件事，记住这条：

> 模型不负责生成坐标。前端先算好对象和锚点，模型只负责读上下文并给出文字。

## 3. 关键名词

### 3.1 Stroke：一笔

定义在 [src/app/state.ts](../src/app/state.ts)：

```ts
export interface Stroke {
  tool: Tool;
  points: StrokePoint[];
}
```

每个点是归一化坐标：

```ts
{ x, y, t, pressure }
```

- `x/y`：相对当前页宽高的 `[0,1]` 坐标。
- `t`：相对落笔时刻的毫秒数。
- `pressure`：笔压，0 到 1。

### 3.2 AnnotationEvent：一笔被打包后的事件

定义在 [src/core/contracts.ts](../src/core/contracts.ts)。它把一笔 stroke 包成标准契约，包含：

- `event_id`
- `trace_id`
- `document_id`
- `page_id`
- `event_type`
- `geometry.bbox`
- `stroke_points`
- `pointer_type`
- `version`

它还是“单笔级别”的数据。

### 3.3 Mark：一处区域里连续画出的一个标注单元

定义在 [src/capture/session.ts](../src/capture/session.ts)。

一个 mark 可能由多笔组成。例如你先圈一个词，又在圈旁边补一个问号，这些笔可能被合成一个 mark。

Mark 里保存：

- 代表事件：`event`
- 几何分类结果：`scored`
- 特征分类结果：`feature`
- 取证记录：`hmp`
- 被标到的文字：`markedText`

### 3.4 HMP：一次手势的取证记录

HMP 是 Hand-Mark Protocol，定义在 [src/core/contracts.ts](../src/core/contracts.ts)，由 [src/evidence/target.ts](../src/evidence/target.ts) 生成。

它回答这些问题：

- 这次标注在哪个 surface 上？
- 这是圈、划线、箭头、手写，还是草图？
- 它锚定了哪些对象？
- 它命中的是文本、图片、空白，还是未知区域？
- 有没有裁图、手写转写、局部 OCR 结果？

HMP 只放事实，不放 AI 推断。

### 3.5 Session：一次 AI 回应前累积的一段标注

Session 是按书累积的，不是按页累积的。翻页不会自动清掉 session。

触发 AI 的主要方式有两个：

- 长停顿：默认 90 秒没有新 mark，提交整段 session。
- 手写：识别成 handwriting 后，先判断这是不是在问 AI，是就提交。

### 3.6 MarkGraph：把一段标注连成关系图

定义在 [src/evidence/mark-graph.ts](../src/evidence/mark-graph.ts)。

图里有三类边：

- 时间边：哪个 mark 先发生，间隔多久。
- 空间边：两个 mark 是否靠近、包含、命中同一对象。
- 语义边：例如箭头从 A 指向 B。

这一步不调用模型，是纯确定性逻辑。

### 3.7 InferenceView：给模型看的精简载荷

定义在 [src/evidence/inference-view.ts](../src/evidence/inference-view.ts)。

它把复杂的 mark graph 蒸馏成模型容易读的内容：

- `narrative`：用户这一阵标注的自然语言叙事。
- `marked`：被标到的文字。
- `page_context`：当前页附近的滑动文本窗口。
- `question`：手写问题。
- `anchor_refs` / `anchor_bbox`：回屏时的锚点。
- `recall`：同页附近历史标注召回。

模型只看这个视图和聊天历史，不直接看底层 stroke 或坐标分数。

### 3.8 ScreenOverlay：屏幕上的 AI 旁注

定义在 [src/core/contracts.ts](../src/core/contracts.ts)。

它是最终渲染对象：

- `display_text`：AI 文字。
- `geometry.anchor_bbox`：锚定位置。
- `object_refs`：跨视图定位用的对象引用。
- `state`：`shown / accepted / edited / dismissed`。

原版页由 [src/surface/whisper.ts](../src/surface/whisper.ts) 渲染，重排页由 [src/surface/reader.ts](../src/surface/reader.ts) 渲染。

## 4. 全局设计原则

### 4.1 全程使用归一化坐标

坐标换算集中在 [src/core/transform.ts](../src/core/transform.ts)：

- `NormBBox = [x, y, w, h]`
- `normToPx`
- `pxToNorm`
- `pageCss`

页面放大、缩小、换设备，只会改变像素尺寸，不会改变归一化坐标。

右侧 AI 留白常量：

```ts
GUTTER_W = 300
GUTTER_PAD = 20
```

### 4.2 笔迹尽量无损

[src/capture/ink.ts](../src/capture/ink.ts) 会优先读 `getCoalescedEvents()`，保留更密集的采样点。

唯一提前过滤的是极小抖动：

```ts
DEADZONE_PX = 1.3
```

它只丢掉和上一点距离小于 1.3 CSS px 的移动，目的是让点按和手抖更稳定地落到 `tap_region`。

### 4.3 先用便宜的确定性逻辑，再用模型

成本顺序是：

```text
几何过滤
→ 文本层命中
→ HMP 取证
→ 按需裁图 / 手写识别 / OCR
→ 关系图和蒸馏
→ 主模型
```

能本地算清楚的，不让模型猜。

### 4.4 模型只引用对象，不吐坐标

PDF 文本层会被拆成字符级 `SurfaceObject`。圈或划命中的不是“字符串猜测”，而是一串对象 id。

AI 回答只产生文字，锚点由前端已有的 `object_refs` 和 `anchor_bbox` 决定。

这是防止模型幻觉 bbox 的关键。

### 4.5 真相源是账本，屏幕状态是派生物

IndexedDB 里最重要的是两本 append-only 账本：

- `marks`：用户标注账本。
- `ai_turns`：AI 回复账本。

擦除不是直接删除，而是追加 tombstone。编辑或忽略旁注不是覆盖原记录，而是追加 supersedes。

刷新页面后，应用从账本折叠恢复当前状态。

## 6. 从导入 PDF 到页面对象表

### 6.1 PDF 加载

文件选择或拖拽后，[src/surface/renderer.ts](../src/surface/renderer.ts) 的 `loadFile(file)` 读取 `ArrayBuffer`。

流程：

1. 计算 PDF hash。
2. 生成稳定 `documentId = doc_ + hash前12位`。
3. 用 PDF.js `getDocument()` 加载 PDF。
4. 读取页数、metadata、outline。
5. 调 `openDoc()` 打开或创建本地持久记录。
6. 把原始 PDF blob 存到 `pdf_blobs`，以后“最近打开”可免重导。
7. 渲染当前页。

### 6.2 PDF.js 字体资源

一些中文 PDF 没有嵌入完整字体或 CMap。项目安装后会运行 [scripts/copy-pdfjs-assets.mjs](../scripts/copy-pdfjs-assets.mjs)，把：

- `node_modules/pdfjs-dist/cmaps`
- `node_modules/pdfjs-dist/standard_fonts`

复制到：

- `public/cmaps`
- `public/standard_fonts`

这两个目录被 `.gitignore` 忽略，是本地生成资产。

### 6.3 页面渲染

`renderPage()` 做这些事：

1. 用 PDF.js 把当前页画到 `#page-layer` canvas。
2. 根据容器宽度和右侧留白计算页面尺寸。
3. 设置 `pageCss.w / pageCss.h`。
4. 抽取文本层 `state.textBlocks`。
5. 扫 PDF operator list，抽取较大的图片区域 `state.imageRegions`。
6. 生成当前页 `pageId`。
7. 用文本层和图片区构建 `SurfaceIndex`。
8. 发 `page:rendered` 事件。

### 6.4 SurfaceIndex：把页面拆成可命中的对象

[src/evidence/target.ts](../src/evidence/target.ts) 的 `wrapSurfaceIndex()` 会把 PDF 文本层拆成字符对象：

```text
文本 run tl_3: "hello"
→ tl_3_0 "h"
→ tl_3_1 "e"
→ tl_3_2 "l"
→ tl_3_3 "l"
→ tl_3_4 "o"
```

每个字符都有自己的 bbox。这样圈一个词时，系统能命中具体字符，而不是只能命中整行。

图片区域也会成为对象：

```text
img_<pageIndex>_<index>
```

这些对象后面会被 HMP 引用。

## 7. 笔迹采集

### 7.1 入口

笔迹输入在 [src/capture/ink.ts](../src/capture/ink.ts)。

初始化发生在 [src/main.ts](../src/main.ts)：

```ts
initInk(document.getElementById('ink-layer'), callback)
```

### 7.2 笔和手指分流

`resolveIntent(pointerType)` 负责判断这是标注还是导航：

- `pen`：标注。
- `touch`：导航。
- 鼠标：看当前工具，如果是 `hand` 就导航，否则标注。

手指横滑超过：

```ts
SWIPE_MIN_PX = 60
```

会触发翻页事件 `nav:flip`。

### 7.3 采样和绘制

`pointerdown` 创建 `live` stroke。

`pointermove`：

1. 优先读取合并事件 `getCoalescedEvents()`。
2. 把像素坐标转成归一化坐标。
3. 小于 `DEADZONE_PX = 1.3` 的移动跳过。
4. 画到 `#ink-layer`。
5. 追加到 `live.points`。

`pointerup`：

1. 把 `live` 打包成 `Stroke`。
2. 加到当前页 `strokesByPage`。
3. 回调给 [src/main.ts](../src/main.ts)。

### 7.4 擦除和撤销

橡皮通过点附近命中已有 stroke。

如果这个 stroke 已经属于某个 mark，擦除会删除整个 mark 的全部 stroke，并发：

```text
mark:erase
```

main 收到后会往 `marks` 账本追加 tombstone。

## 8. 单笔事件和区域组装

### 8.1 recordEvent

[src/core/pipeline.ts](../src/core/pipeline.ts) 的 `recordEvent()` 把 Stroke 变成 AnnotationEvent。

它会：

1. 调 `makeEvent()`。
2. 算 stroke bbox。
3. 初步几何分类。
4. 写 trace。
5. 记录 pen-up 到 event 的延迟指标。

### 8.2 为什么还要区域组装

用户的一次表达常常不是一笔。例如：

- 圈一个词，再写一个问号。
- 划一行，再在旁边写“why”。
- 写一个中文短句，由多笔组成。

所以 main 里维护当前“活跃区域”：

```ts
regEvents
regStrokes
regBbox
regTimer
```

### 8.3 区域规则

核心参数在 [src/main.ts](../src/main.ts)：

```ts
REGION_QUIET_MS = 6000
REGION_NEAR = 0.06
```

规则：

- 新笔在当前区域 bbox 外扩 `0.06` 内，合并到当前区域。
- 新笔落到远处，先收口旧区域，再开启新区。
- 附近 6 秒无动作，收口当前区域。

收口后进入 `resolveRegion()`。

## 9. 几何分类

分类代码在 [src/capture/classify.ts](../src/capture/classify.ts)。

### 9.1 第一层：像不像一个几何手势

`classifyScored(points, bbox)` 输出：

```ts
{ type, score, raw, arrow? }
```

支持：

- `tap_region`
- `circle`
- `underline`
- `arrow`
- `stroke`

几个重要判断：

- 总行程太短，或点数太少，判 `tap_region`。
- 圈：起点终点接近，路径绕得够长。
- 划线：宽高比大，路径接近直线。
- 箭头：非闭合，主干较直，端部有急转。
- 小于约 3mm 的圈/划会降级为自由笔。

### 9.2 第二层：这是标记，手写，还是画

`classifyStrokeFeature()` 把一组笔分成：

- `markup`：干净的圈、划线、箭头。
- `handwriting`：最终由识别接口确认。
- `drawing`：画、线条、草图或无法识别的自由笔。

这里有一个重要设计：

> 几何只可靠识别“圈/划/箭头”这类模板手势；自由笔到底是字还是画，交给识别模型定型。

自由笔如果满足：

- 不太小。
- 多笔，或单笔路径足够复杂。

就会标记 `ocrWorthy = true`，后面才调用 `/api/interpret`。

## 10. captureMark：落笔当时完成取证

`captureMark()` 在 [src/core/pipeline.ts](../src/core/pipeline.ts)。

为什么必须在落笔当时取证？

因为 session 可跨页。等到 90 秒后提交时，用户可能已经翻页了，当前 canvas 和文本层已经不是那一页。那时再裁图或查文本层就错了。

所以每个 mark 在生成时就保存：

- HMP。
- markedText。
- 必要的 crop / vector。
- 识别阶段 trace。

### 10.1 目标命中

`resolveTarget(events, bbox, index)` 做两级命中：

1. 包围：对象中心点是否在闭合笔迹多边形内。
2. 相交：mark bbox 外扩后，与对象 bbox 是否有足够重叠。

命中对象按阅读顺序排序。

### 10.2 生成 HMP

`buildHmp()` 生成取证记录。

HMP 的 `mode`：

- `anchored`：命中了文本或图片等内容对象。
- `self_content`：手写或画本身就是内容。
- `mixed`：同时命中空白和内容。
- `unknown`：暂时无法判断。

HMP 的 `object_hint`：

- `text`
- `image_region`
- `ui_region`
- `blank`
- `diagram`
- `unknown`

### 10.3 自由笔识别

如果自由笔 `ocrWorthy = true`，会从 canvas 裁出白底笔迹图：

```ts
grabLayers(bbox).ink
```

然后调用：

```text
POST /api/interpret
```

服务端使用 `ink_classifier` prompt，输出：

```json
{"kind":"handwriting|sketch|mixed|none","reading":"...","description":"..."}
```

前端再把 feature 定型为 handwriting 或 drawing。

### 10.4 图片和无文字区域 OCR

如果标注命中图片，或命中没有文字的对象，会裁合成图：

```ts
grabRegion()
```

并调用：

```text
POST /api/ocr-vlm
```

读出的文字填到 `hmp.text_hint`。

## 11. Session 提交

### 11.1 什么时候提交

提交函数是 [src/core/pipeline.ts](../src/core/pipeline.ts) 的 `commitSessionDiscussion()`。

由 [src/main.ts](../src/main.ts) 的 `commitSession()` 调用。

触发方式：

- 手写：一旦 mark 被定型为 `handwriting`，立即进入上下文分类器。
- idle：长停顿后提交整段 session。

默认长停顿：

```ts
IDLE_COMMIT_MS = 90000
```

设置页可改 `settings.gesture.idleSeconds`。

### 11.2 手写不是一定回答

手写触发后，系统会先调用：

```text
POST /api/classify-context
```

判断：

- `respond=true`：这是问 AI 或命令 AI，继续主模型。
- `respond=false`：这是写给自己的笔记，不打扰。

网络或解析失败时默认 `respond=true`，因为漏答真问题更贵。

### 11.3 idle 提交会过滤无意义涂写

idle 提交前会检查 session 是否有实质内容：

- 有手写。
- 或锚定到真实正文。
- 或画被识别成有描述的草图。

纯散笔、误触、无内容涂鸦会被清掉，不触发 AI。

## 12. MarkGraph：确定性关系建模

关系图在 [src/evidence/mark-graph.ts](../src/evidence/mark-graph.ts)。

### 12.1 时间边

相邻 mark 会连一条时间边：

```text
prev -> cur
```

时间近阈值：

```ts
NEAR_TIME_MS = 30000
```

burst gap：

```ts
BURST_GAP_MS = 12000
```

### 12.2 空间边

所有 mark 两两比较：

- 包含：一个 mark 的闭合路径包住另一个 mark 的中心。
- 邻近：中心距离小于 `SPATIAL_NEAR = 0.12`。
- 同目标：两者命中了同一个 object ref。

### 12.3 四象限

时间和空间组合成四种关系：

| 时间 | 空间 | 标签 | 含义 |
|---|---|---|---|
| 近 | 近 | `one_action` | 一口气做的 |
| 近 | 远 | `sweep` | 扫读到另一处 |
| 远 | 近 | `revisit` | 回到之前那处 |
| 远 | 远 | `separate` | 另起一件事 |

### 12.4 箭头语义边

箭头会根据笔迹急转位置判断 tip 端，再把 tail 最近的 mark 指向 tip 最近的 mark。

## 13. 空间召回：找回附近历史标注

每次 AI 回复成功后，当前 session 会清空。这样能避免重复综合，但也带来一个问题：

> 旁边明明有旧标注，本轮 mark graph 看不到。

[src/evidence/recall.ts](../src/evidence/recall.ts) 会从 `marks` 账本里找同页、同区域附近的旧 mark。

规则：

- 只查同一页。
- 不把旧 mark 放进当前 graph nodes。
- 最多召回 `RECALL_K = 3` 条。
- 支持中心距离近、圈住、同行边注三条通道。

同行边注参数：

```ts
ROW_BAND = 0.03
ROW_REACH = 0.5
```

召回结果会进 InferenceView 的 `recall`，让模型知道“这附近你之前也标过什么”。

## 14. InferenceView：把复杂数据变成人话

[src/evidence/inference-view.ts](../src/evidence/inference-view.ts) 做一件事：

> 把 mark graph 投影成一段模型能读懂的叙事。

例子：

```text
圈「A」，随即划线「B」；位置关系上，「A」与「B」紧挨在一处
```

它会生成：

- `narrative`
- `marked`
- `page_context`
- `question`
- `crop`
- `anchor_refs`
- `anchor_bbox`
- `referent_lines`
- `recall`
- `thematic`

注意：

- `page_context` 是当前页为中心的滑动窗口，约 3000 字。
- 当前页如果有重排缓存，会优先用重排后的阅读顺序。
- 手写问题会尝试取同一纵向带的印刷正文行，填 `referent_lines`，解决“这个是什么意思”里的“这个”。

## 15. AI 调用和服务端代理

### 15.1 为什么有 server 目录

前端不能直接带 key 调模型。Vite dev server 通过 [vite.config.ts](../vite.config.ts) 注册 `/api/*` 中间件，请求进 Node 侧，再由 [server/infer.ts](../server/infer.ts) 调 NoDesk AI Gateway。

Key 来自 `.env`：

```text
LLM_GATEWAY_URL
LLM_GATEWAY_KEY
LLM_MODEL
```

### 15.2 模型路由

[server/infer.ts](../server/infer.ts) 的规则：

- `kimi*` → `channel = kimi`
- 其他模型 → `channel = DMX`

前端多数请求会显式传：

- `settings.inferModel`
- `settings.reflowModel`

所以 `.env` 里的 `LLM_MODEL` 更多是服务端兜底。

当前默认：

```ts
settings.inferModel = 'claude-sonnet-4-6'
settings.reflowModel = 'gemini-3.1-flash-lite'
```

### 15.3 Prompt 注册表

所有 system prompt 在 [server/prompts.ts](../server/prompts.ts)：

- `annotator`：主伴读。
- `ink_classifier`：手写 vs 画、转写、草图描述。
- `context_classifier`：手写 respond/fold。
- `ocr`：OCR 转写。
- `image_explain`：图片解读。
- `reflow_refine`：重排精修。
- `reflow_structure`：文本驱动结构重建。
- `reflow_vlm`：看图重排。

## 16. AI 回屏

### 16.1 流式锚点预览

主模型流式输出时，[src/chat/stream-client.ts](../src/chat/stream-client.ts) 的 `onDelta` 会发：

```text
anchor:place
```

[src/surface/anchor-layer.ts](../src/surface/anchor-layer.ts) 根据 `anchor_refs` 或 bbox 先把正在生成的文本贴到页面上。

模型完成后发：

```text
anchor:clear
```

然后生成正式 overlay。

### 16.2 原版页面旁注

[src/surface/whisper.ts](../src/surface/whisper.ts) 负责原版页面。

两种落点：

- `margin`：右侧留白，按锚点 y 排序，重叠就下推。
- `inline`：贴在正文旁边，右边放不下就放左边，再放不下就放下方。

默认是：

```ts
settings.placement = 'margin'
```

旁注支持：

- 收下：`accepted`
- 改写：`edited`
- 散去：`dismissed`

状态变化会写入 `ai_turns` 的 supersedes 记录。

### 16.3 重排阅读视图旁注

[src/surface/reader.ts](../src/surface/reader.ts) 负责重排视图。

它先把原 PDF 文本重排成块，然后建立：

```text
字符对象 id → 重排块 id
```

overlay 里如果有 `object_refs`，就能找到对应重排块，把旁注放在该块右侧。

如果 ref 缺失或旧缓存不完整，就按 `anchor_bbox` 的 y 中心找最近块兜底。

## 17. 重排阅读

### 17.1 本地重排

[src/surface/reflow.ts](../src/surface/reflow.ts) 的 `reflowLocal()` 是纯几何启发式：

1. 剥页眉页脚。
2. 按 y 聚行。
3. 行内按 x 排序。
4. 按字号识别标题。
5. 按段间距聚段。
6. 识别简单列表。

它适合单栏数字 PDF，不适合复杂多栏、表格、公式、网页截图 PDF。

### 17.2 AI 重排 provider

[src/surface/reflow-provider.ts](../src/surface/reflow-provider.ts) 提供多种 provider：

- `ai`：主线，行级文本给模型分组成 heading / para / list，保 bbox。
- `local`：纯本地启发式。
- `hybrid`：本地块给模型精修文字和类型。
- `vision`：本地块加页面图给模型调整顺序。
- `rewrite`：VLM 看整页图直接重写块，bbox 是模型估计，跨视图锚点不精确。

默认：

```ts
settings.reflowProvider = 'ai'
```

### 17.3 重排视图里也能圈画

重排页不是只读渲染。

当用户在重排段落上画笔迹时，reader 会：

1. 命中被画到的重排块。
2. 把内容坐标映射回这个块的原 PDF bbox。
3. 生成 `pointer_type = 'reader'` 的 AnnotationEvent。
4. 发 `reader:gesture`。
5. main 走同一套 `resolveRegion → captureMark → session → AI` 管线。

所以原版页和重排页共用同一套 mark 账本。

## 18. IndexedDB 持久化

持久化在 [src/local/store.ts](../src/local/store.ts)，格式在 [src/core/store-format.ts](../src/core/store-format.ts)。

数据库名：

```text
inkloop
```

版本：

```ts
DB_VERSION = 3
STORE_VERSION = '2'
```

### 18.1 四个 store

| Store | 作用 | 是否真相源 |
|---|---|---|
| `docs` | 书目元信息、页缓存、阅读位置、水位线 | 部分是缓存 |
| `pdf_blobs` | 原始 PDF 字节 | 是 |
| `marks` | 用户标注账本 | 是 |
| `ai_turns` | AI 回复账本 | 是 |

### 18.2 marks 账本

每个 mark 追加一条 `PersistedMark`：

- `mark_id`
- `strokes`
- `bbox`
- `feature_type`
- `hmp`
- `marked_text`
- `is_tombstone`

擦除时追加 tombstone，不直接删除旧记录。

`getFoldedMarks()` 会折叠账本，得到当前仍然有效的 mark。

### 18.3 ai_turns 账本

每次 AI 回复追加一条 `PersistedAiTurn`：

- `overlay`
- `ai_reply`
- `thinking`
- `anchor`
- `inference_view`
- `prompt_snapshot`
- `system_prompt_hash`
- `settings_snapshot`
- `pipeline`
- `supersedes`

旁注被接受、编辑、忽略时，会追加一条新的 ai_turn supersede 旧条目。

### 18.4 刷新后如何恢复

[src/main.ts](../src/main.ts) 的 `restoreFromLedger()` 做恢复：

1. 从 `marks` 折叠出当前笔迹，重建 `strokesByPage`。
2. 从 `ai_turns` 折叠出当前 overlay。
3. 把最近 3 轮 AI 对话回放进 chat buffer。
4. 找水位线后的 pending marks，重建内存 session。
5. 发 `page:rendered` 让画布和旁注重绘。

## 21. 已知边界

### 21.1 数字 PDF 优先

文本命中依赖 PDF 文本层。数字版 PDF 效果最好。

扫描版 PDF 没有可靠文本层，只能依赖 OCR / VLM。没 key 时无法准确知道圈住了什么字。

### 21.2 重排不是还原版式

重排视图追求阅读顺序和段落结构，不保留原版视觉排版。

需要看原始版式时，切回「原版」。

### 21.3 本地几何分类只管便宜可靠的事

圈、划、箭头能用几何判断。自由手写和草图不强行本地猜，最终要靠识别模型。

### 21.4 当前是 Web demo

项目验证的是数据契约、交互节奏和回屏方式。它还不是电子纸硬件端实现。

不过因为所有几何都用归一化坐标，后续迁移运行时不会推翻数据链路。

## 23. 最小心智模型

最后用最短的话再串一次：

1. PDF 渲染时，前端把页面拆成很多可命中的文字/图片对象。
2. 用户画笔迹时，前端保存完整点序，并过滤误触。
3. 多笔会先在空间和时间上合成一个 mark。
4. mark 立刻做 HMP 取证，记录它标到了什么。
5. 多个 mark 累积成 session。
6. 手写问题或长停顿让 session 提交。
7. session 先变成 mark graph，再变成 InferenceView。
8. 模型只读 InferenceView 和少量聊天历史。
9. 模型回的文字被前端贴回已知锚点。
10. 用户笔迹和 AI 回复都进 append-only 账本，刷新后从账本恢复。
