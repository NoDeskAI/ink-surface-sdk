# InkLoop 要做跨端，但不能把每个端做成同一个东西

这份演示稿用于讨论 InkLoop 在墨水屏一体机之外，是否应该覆盖桌面端、移动端和 Web 端；以及每个端各自承担什么角色。

结论：跨端有必要，但 v1 应该先做“跨设备工作流”，而不是“全平台全功能原生客户端”。

InkLoop 的核心资产不是某一块屏幕，而是用户在文档上的标注、笔迹、AI 旁注、上下文和知识账本。设备只是不同入口。

手机是入口，桌面是工作台，墨水屏是深度思考终端，Web 是低摩擦入口和分享层。

1第一天就要跨端的数据：文档、标注、AI 回答、阅读进度、索引和账本。

4四个端的定位必须不同：收集、整理、深读、分享。

Strategic frame

## 先把“跨平台”拆成两件事

很多团队会把跨平台理解为“每个系统都做一个完整 App”。对 InkLoop 来说，更正确的拆法是：核心工作流必须跨端，客户端形态可分阶段。

先统一数据，再扩展端

必须第一天设计
### 跨端的是数据与工作流

文档库、文本/OCR 层、笔迹、标注对象、AI 旁注、问答历史、阅读进度、索引和用户知识上下文，需要从第一天就按跨设备系统设计。

不必第一天铺满
### 客户端不必全平台全功能

v1 不采用同时重兵投入 Windows、macOS、Linux、iOS、Android 和 Web 的完整体验。要先做出不同端的核心价值，而不是平均撒胡椒面。

核心判断：InkLoop 应该是“跨设备的 AI 阅读/标注/思考系统”，不是单纯的墨水屏硬件 App。

Device roles

## 四个端的定位

每个端都应该承担最适合自己的任务。端之间是接力关系，不是替代关系。

收 → 整 → 想 → 分享

手机端
### 资料漏斗

不是深度阅读主战场，而是随手收集入口。

- Share Sheet 导入网页/PDF/附件

- 拍照扫描、截图保存、语音备注

- 待读队列、快速摘要、碎片复看

桌面端
### 深度工作台

承担大屏、多文档、键盘和文件系统优势。

- 文档导入、批量管理、文件夹监听

- 多文档研究、引用、整理和输出

- 软件-only 试用和转化漏斗

墨水屏
### 深度思考终端

旗舰体验端，负责长阅读、手写和低干扰。

- 纸感阅读、手写标注、低通知干扰

- AI 回答贴回原文现场

- 弱网/离线下仍可做端侧取证

Web 端
### 低摩擦入口

适合试用、分享、临时访问和轻协作。

- 打开即用、上传 PDF、账号订阅

- 分享 AI 旁注文档给同事/客户/课堂

- 轻阅读、轻问答、管理设备

Platform priority

## 这些平台都需要吗？需要，但优先级不同

长期看桌面、移动和 Web 都有价值；短期要按商业价值、实现复杂度和端定位排序。

v1 不做大一统

平台要求优先级v1 定位为什么

墨水屏一体机最高旗舰体验端最能体现纸感、专注、手写标注和 AI 回屏，是品牌心智与硬件溢价来源。

macOS高桌面工作台 / 软件-only知识工作者、研究者、产品/设计/创业用户密度高，适合深度整理和输出。

Windows高桌面工作台 / 企业教育入口企业、教育、法律、金融、工程用户多；Surface/触控笔场景也自然。

Linux低到中先用 Web/PWA 覆盖用户可能很喜欢，但商业优先级不应排在前面；原生客户端可后置。

iOS / iPadOS高Capture companion；iPad 可做软件-only美国 C 端、学生、研究者、知识工作者中 iPhone/iPad 密度高；Share Sheet 价值极大。

Android中到高Capture companion；与硬件体系衔接全球覆盖广，且墨水屏设备大概率是 Android/RK 体系，长期必须支持。

Web高试用、上传、分享、账号、轻阅读获客和协作阻力最低，也能兜底 Linux 和临时设备访问。

要求排序：墨水屏端 ＞ 桌面/Web ＞ iOS companion ＞ Android companion ＞ Linux 原生。

Scenario value

## 你提到的几个使用场景，都有价值

关键不是“这个端能不能做”，而是“它在整个知识流里承担什么位置”。

场景成立，但边界要清楚

场景 A
### 买一体方案：墨水屏 + 笔 + 软件

这是品牌旗舰和最高差异化 SKU。卖点不是“又一个电子书阅读器”，而是“能理解你标注意图的 AI 纸面工作台”。

场景 B
### 桌面端用自己的笔 + 板子 + 软件

值得做成 InkLoop Studio Mode。它降低试用门槛，也能让硬件量产前先验证留存和工作流。

场景 C
### 手机上收集文档

非常有价值。移动端不抢深度阅读，而是做资料收件箱：分享、扫描、截图、轻摘要、稍后阅读。

场景 D
### 办公室 / 家里 / 路上连续工作

这是跨端的核心体验：桌面导入和整理，墨水屏深读和标注，手机补充收集，Web 分享给别人。

Product packaging

## 要求拆成四个产品形态

它们可共享账号、文档库、AI 能力和同步协议，但对外表达要清晰。

硬件旗舰 + 软件漏斗

01

### InkLoop Paper

墨水屏 + 笔 + 本地取证 + 云端高质量答问 + 跨端资料库。主打专注阅读、手写标注和 AI 回屏。

02

### InkLoop Studio

桌面/平板软件-only。支持 PDF 导入、手写/鼠标标注、多文档问答、导出和未来同步到 Paper。

03

### InkLoop Capture

手机 companion。最好免费包含在账号里，主要负责分享导入、扫描、待读队列和快速摘要。

04

### InkLoop Web

轻入口、分享页、账号订阅、团队/课堂/客户访问。不是最高保真手写主战场。

What actually syncs

## 跨端真正同步的不是“文件”，而是“思考现场”

普通产品同步 PDF 和笔记图片就结束了；InkLoop 应该同步用户与原文互动后产生的一整套结构化上下文。

文件只是底座

原始文档PDF、网页、扫描件、图片、附件，确保任何端都能打开。

文本 / OCR 层支撑搜索、问答、引用和跨视图锚定。

笔迹 Stroke保留用户原始手写表达，而不是只保存截图。

标注对象 ID跨设备重新定位用户圈画的字、图、段落和区域。

HMP / MarkGraph保留用户标注时的事实取证与关系结构。

InferenceView给模型看的精简上下文，让各端 AI 行为一致。

AI 旁注与问答形成可复看的思考记录，而不是一次性聊天。

阅读进度桌面导入，墨水屏继续读，手机碎片复看。

向量索引 / 召回缓存支撑 Ask my notes、Ask my library 和多文档研究。

操作账本支持冲突合并、撤销、刷新恢复和跨端一致性。

用户知识上下文把单篇阅读延伸成长期知识库。

设备状态离线、弱网、同步水位线和 pending 操作都要可解释。

当前前端链路已经有天然跨端基础：归一化坐标、对象引用、marks / ai_turns append-only 账本、HMP、MarkGraph 和 InferenceView。这些都应该被抽象成 InkLoop Core。

InkLoop Core

## 要求形成一个平台无关的核心层

不同端不要各写一套“AI 标注逻辑”。应尽量把跨端一致的东西收敛到 Core。

Core + Shells

### InkLoop Core 应该包含

- 文档对象层与页面对象 ID

- 笔迹 / 标注账本

- HMP 取证

- MarkGraph 与 InferenceView

- AI 路由与回屏锚点

- 同步协议、冲突处理和权限模型

### 各端只是不同 Shell

- 墨水屏 Shell：EPD、笔、弱网、端侧 OCR/索引

- 桌面 Shell：文件系统、多窗口、键盘、导出

- 手机 Shell：Share Sheet、相机、扫描、通知

- Web Shell：分享、试用、账号、轻阅读

Boundaries

## 最容易踩的坑

跨端真正的风险不是“端太少”，而是“每个端都想做完整”。

边界比功能更重要

应该坚持
### ✅ 做跨端工作流

- 手机收集，桌面整理，墨水屏深读，Web 分享。

- 从第一天统一文档对象、标注账本、AI 旁注和同步协议。

- 桌面端作为软件-only 漏斗，但不替代一体机。

- Web 做试用和分享，不强行承担最高保真手写。

不要过早做
### 🚫 做全平台同构 UI

- 不要让手机端承担完整 PDF 手写标注。

- 不要 v1 就重兵投入 Linux 原生客户端。

- 不要把 Web 做成复杂专业手写主战场。

- 不要让每个平台都有同一套完整功能。

## 一句话要求

InkLoop 应该卖硬件，但不能只活在硬件里；应该跨平台，但不能把每个平台做成同一个东西。
